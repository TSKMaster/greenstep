# GreenStep Project Context

## Что это за проект

`GreenStep` — конкурсный `MVP` web-приложения для жителей района.

Идея проекта: пользователь может сообщать об экологических проблемах, видеть обращения на карте, следить за статусами и участвовать в жизни района.

Проект делается как `mobile-first web MVP`, но должен оставаться удобным и на десктопе.

## Стек

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- `Supabase`
- `Leaflet`

## Что уже реализовано

- вход через `magic link`
- `Resend` подключен как custom SMTP для auth-писем через `Supabase`
- кастомные email-шаблоны для auth-писем
- создание заявки
- выбор точки на карте при создании заявки
- загрузка фото к заявке
- общий список заявок
- список моих заявок
- карточка отдельной заявки
- карта заявок
- локализованные статусы в popup карты
- базовая статистика
- админка
- отдельная страница редактирования заявки в админке
- защита от повторной поддержки заявки
- запрет поддержки собственной заявки
- клиентская валидация формы создания заявки
- понятный `error handling` для входа, отправки заявки, поддержки заявки и действий в админке
- исправлен mobile auth flow для локальной сети
- исправлен mobile submit flow для создания заявки
- проект проходит `npm run lint`
- проект проходит `npm run build`

## Текущий этап

Сейчас основной фокус не на большом редизайне, а на доведении `MVP` до надежного и понятного состояния.

Что уже подтверждено вручную:

- базовый авторизованный desktop smoke-check по логике
- вход через `magic link` на реальном телефоне в локальной сети
- создание заявки с телефона, включая фото
- базовая карта, списки и статистика открываются и работают по логике

Что сознательно отложено:

- широкая полировка мобильного интерфейса
- большой UI-редизайн
- приведение экранов ближе к макетам

Ближайшие рекомендуемые задачи:

- добить цельный desktop smoke-check как полный пользовательский сценарий
- добить полный мобильный smoke-check по оставшимся разделам, когда вернемся к mobile UI
- продолжить полировку текстов и кодировки

## Ближайшие приоритеты

### Критично для конкурса

- протестировать проект на реальном телефоне как цельный сценарий
- протестировать проект на десктопе как цельный сценарий

### Важно, но после этого

- написать 2-3 простых unit-теста для ключевой логики
- покрыть тестами расчет `эко-индекса`
- покрыть тестами логику поддержки заявки
- добить цельность `UX`: пустые состояния, тексты, понятные переходы

### Если останется время

- подумать про пагинацию списков
- подумать про кэширование тяжелых запросов
- оценить, нужны ли дополнительные провайдеры

## Важные договоренности

- пока не делать большой `UI`-редизайн без явной необходимости
- `mobile-first` важен всегда
- на десктопе потом можно делать отдельную визуальную подачу без ломки логики
- если нужен `SQL` для `Supabase`, писать его прямо в чат, а не создавать отдельный файл под одноразовый запуск
- маленькие usability-фиксы допустимы даже до этапа большого UI
- для auth-писем используем `Resend` как основной SMTP-провайдер вместо встроенной почты `Supabase`
- `TODO.md`, `PROJECT_CONTEXT.md` и `README.md` нужно обновлять после каждой заметной завершенной задачи
- `MCP` и `skills` используем только когда они реально помогают текущей задаче

## Работа с инструментами

Для новых задач сначала проверять, могут ли помочь подключенные `MCP` и доступные `skills`.

Правило работы:

- сначала оценить, есть ли подходящий `MCP` или `skill` под текущую задачу
- если инструмент реально помогает, использовать его
- если он не помогает или только усложняет работу, решать задачу обычным способом
- не использовать инструменты формально, только ради пользы для проекта

## Что нужно держать актуальным

- `README.md` — если меняется функциональность, структура проекта или запуск
- `TODO.md` — если меняются задачи и приоритеты
- `DATABASE_SIMPLE.md` — если меняется база данных
- `PROJECT_CONTEXT.md` — если меняется этап проекта, договоренности или важные решения

## Что пока сознательно не делаем

- не уходим глубоко в архитектурные усложнения
- не делаем тяжелую оптимизацию запросов без реальной необходимости
- не делаем полноценный большой редизайн до закрытия основных `MVP`-рисков

## Preview desktop redesign

Текущее состояние по редизайну главной страницы:

- основная главная страница проекта по-прежнему живёт в `src/app/page.tsx` и не заменяется preview-версией
- отдельный desktop preview собран в маршруте `/preview/main-redesign`
- preview используется как безопасная зона для UI/UX-итераций без риска сломать текущую MVP-логику
- preview использует реальные данные заявок, реальный расчёт эко-индекса и реальную карту на `Leaflet`
- недостающие разделы в preview допускаются как demo-заглушки с сообщением, что раздел находится в разработке

Текущие файлы preview-версии:

- `src/app/preview/main-redesign/page.tsx`
- `src/components/home-preview/main-redesign-preview.tsx`
- `src/components/home-preview/preview-reports-map.tsx`
- `src/components/home-preview/preview-reports-map-loader.tsx`

Рабочие договорённости по preview:

- сначала доводим desktop preview до визуально согласованного состояния
- только после этого возвращаемся к mobile-адаптации
- после каждой заметной визуальной правки preview обязательно проверяется в браузере и подтверждается скриншотом
- до отдельного решения изменения по редизайну главной страницы вносятся только в preview, а не в основную главную
- текущий восстановленный рабочий проект находится в папке `greenstep` и именно она считается основной после восстановления из GitHub
