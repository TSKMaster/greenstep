# GreenStep Project Context

## Что это за проект

`GreenStep` — конкурсный `MVP` веб-приложения для жителей района.

Базовая идея:

- пользователь авторизуется через `magic link`
- отмечает экологическую проблему на карте
- прикладывает описание и фото
- отправляет обращение
- видит обращения на карте и в списках
- отслеживает статус своей заявки

Ключевая рамка проекта:

- это именно конкурсный `MVP`, а не полнофункциональный городской сервис
- рабочее ядро сейчас: авторизация, создание заявки, просмотр карты и списка заявок
- остальные разделы могут быть demo-сценариями, если это честно показано в интерфейсе

## Текущий стек

- `Next.js`
- `TypeScript`
- `Tailwind CSS`
- `Supabase`
- `Leaflet`

## Что уже реализовано

### Основной пользовательский путь

- вход через `magic link`
- страница входа в обновлённом стиле проекта
- подтверждение входа и страница успешной авторизации
- поле имени на входе с сохранением имени пользователя в `profiles.full_name`
- рабочий выход из аккаунта через меню пользователя в хедере

### Работа с заявками

- создание новой заявки
- выбор точки на карте
- автоподстановка адреса по выбранной точке
- загрузка фото
- анонимная отправка
- обязательные шаги формы с визуальной отметкой выполнения
- предупреждение перед отправкой, если обязательные шаги не завершены

### Просмотр заявок

- главная страница уже переведена на новый redesign
- отдельная страница `Карта обращений`
- отдельная страница `Все заявки`
- единый `header + nav` на основных страницах
- адаптивная навигация: на мобильных уходит вниз экрана
- карта на главной и отдельная карта используют одинаковую логику popup и кластеризации
- цветные маркеры по категориям заявок
- интерактивная легенда карты с количеством заявок по категориям
- включение и выключение категорий через легенду
- popup карты со статусом, поддержкой и переходом в карточку заявки

### Страница списка заявок

- счётчики: всего, новых, моих, решено моих
- фильтры по статусам с количеством
- фильтрация списка по нажатию на статус
- пастельные цвета статусов
- CTA `Новая заявка`
- мобильная `FAB`-кнопка для создания заявки

### Страница создания заявки

- desktop-раскладка блоков без ломки существующей логики
- общий хедер и навигация как на остальных страницах
- карта и описание разнесены по шагам
- шаг 1 и шаг 2 обязательны
- шаг 4 оформлен как отдельный блок подтверждения и отправки

## Карта проекта по экранам

- `/` — новая главная страница
- `/map` — карта обращений
- `/reports` — все заявки
- `/reports/new` — создание заявки
- `/profile` — профиль
- `/preview/main-redesign` — preview-маршрут для безопасных UI-экспериментов

## Preview и redesign

Preview-маршрут по-прежнему сохранён:

- `/preview/main-redesign?mode=guest`
- `/preview/main-redesign?mode=auth`

Но важное текущее состояние проекта такое:

- основная главная страница уже переведена на новый redesign
- preview по-прежнему остаётся безопасной зоной для дальнейших UI-экспериментов
- guest/auth toggle нужен только для preview и в продовую главную не переносится

## База данных

Основные таблицы:

- `profiles`
- `reports`
- `report_comments`
- `report_supports`

Важно:

- фото заявок лежат в `Supabase Storage` в bucket `report-photos`
- имя пользователя теперь используется из `profiles.full_name`
- если меняется схема базы, нужно обновлять `DATABASE_SIMPLE.md`

## Auth и сессии

Что уже сделано по стабильности:

- browser Supabase client переведён в singleton
- добавлен middleware для обновления сессии
- в хедере есть выпадающее меню пользователя
- доступны пункты `Профиль` и `Выход`

## Текущие договорённости

- основную логику не ломаем ради косметических правок
- крупные UI-изменения сначала делаем аккуратно и с понятной точкой отката
- после заметных визуальных правок желательно проверять экран в браузере
- `README.md`, `TODO.md` и `PROJECT_CONTEXT.md` нужно обновлять после заметных завершённых задач
